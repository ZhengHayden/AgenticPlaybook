Agentic Workflow
Delivery Playbook

A four-phase methodology — Impact Sizing, Design, MVP, Production — that turns enterprise workflows into validated agentic operations within 3–4 months, with codified frameworks, decision gates, and templates that make delivery repeatable across consultants and clients.

Audience Functional consultants · Agentic architects · Engineers

Reading guide02 · Front matter

This playbook closes the gap between an activity description and an executable engagement — pick up the templates, run the gates, ship the deliverables

Read sequentially for first engagements; jump to a phase index for repeat use

Why it exists
  • 01  Codify a delivery methodology that a 2-person team can execute end-to-end
  • 02  Reduce quality variance across consultants by replacing judgment with rubrics
  • 03  Create defensible competitive positioning vs. generic "AI consulting"
How to use it
  • A  Send Phase-N data request to client 5 business days before kickoff
  • B  Walk the daily agenda; every activity has a lead, support, and outcome
  • C  Produce the sample deliverable; do not over-engineer beyond template
  • D  Pass the phase gate before any spend on the next phase
Scope boundaries
  • In scope  Single-workflow delivery, platform-agnostic frameworks, role definitions
  • In scope  3 interaction archetypes · 5 agent archetypes · 6 A2A patterns · HITL taxonomy
  • Out  Enterprise-wide agent-native org redesign (separate playbook)
  • Out  Vendor selection, commercial model, post-go-live managed service

Overall approach03 · Lead page

Four phases — Impact Sizing → Design → MVP → Production — move an enterprise workflow from candidate to value over 13–17 weeks with progressively narrowing scope and tightening evidence

Each phase has a binding gate; failure to pass returns the workflow to a prior phase or to the next-wave roadmap

Phase 1 · Impact sizing1.2 · Week-0 daily agenda

Five days, three layers — Binary Screen → 2×2 Funnel → Depth Scoring — converge from longlist of 10–20 to a ranked portfolio with quadrant-specific action paths

Layered design concentrates effort on survivors; Day 1 eliminates 30–40% before scoring time is spent

Day Headline activity Lead Support Outcome / Artifact
Day 1
19% time		 Binary readiness screen Functional consultant Agentic architect + Client PMO 30-min interviews per workflow; 6 binary criteria scored; shortlist of 6–10 workflows ≥5/6 Yes documented
Day 2
14% time		 ODS scoring — Output Determinism Functional consultant Process owners + QA leads 4 indicators × 4 weights per workflow; ODS total 0–2; threshold 1.2; ODS scoring table completed
Day 3
14% time		 ORS scoring — Organizational Readiness Functional consultant Sponsors + Change/HR + IT leads 4 indicators (sponsor, team, integration, change history); ORS total 0–2; quadrant assignment for each workflow
Day 4
19% time		 Depth scoring on Q1 / Q2 only Agentic architect Functional consultant + Finance VM (4 dims), DDI, Risk (4 cats); RAS & PriorityScore computed; complete depth-scoring worksheet
Day 5
22% time		 Portfolio synthesis & sponsor read-out Agentic architect Functional consultant 1–2 year wave roadmap; quadrant action paths; sponsor read-out deck + go/park decisions logged
Cross-day cadences
Daily Standup & calibration Both team members — 15 min start-of-day; inter-rater check; flag scoring disagreements >1 pt (target Cohen's κ ≥ 0.70)
Day 3 pm Mid-week sponsor checkpoint Functional consultant Sponsor 30-min sync; preview quadrant placements; align on which Q1/Q2 enter depth scoring
Time budget guardrail ·   Layer 1 (~19%) → Layer 2 (~28%) → Layer 3 (~31%) → Deliverable (~22%). If Layer 2 runs over, deprioritize the lowest-screen survivors rather than compressing Layer 3 scoring rigor.

Phase 1 · Deep-dive1.4 · Binary readiness screen

Six binary criteria — 5-of-6 Yes required — eliminate structurally unready candidates before any time is spent on quantitative scoring

One exception is allowed if mitigation plan is documented; "Maria-only-knows" tribal knowledge is automatic fail

#CriterionPass testWhy it matters
C1DocumentabilityWalkable end-to-end in <2hr OR SOP <12mo old covering ≥80% logicTribal knowledge → Design phase overrun
C2Data accessibilityInput data accessible via API / DB / structured exportPaper or manual data → separate digitization project required
C3Volume>50 executions / month OR >8hr per exec with high error costLow volume → ROI insufficient in 3–4 month window
C4Process ownerNamed owner with authority + 4h/wk + 48-hr decisionsNo owner → Design cannot validate, MVP cannot get data, Prod fails adoption
C5Definable outputQuality definable by rule OR ≥100 golden examples availableNo correctness model → MVP cannot be tested
C6Process stabilityNo major change planned in next 6 monthsMoving target → architecture invalidated before production

Interview protocol (30 min / workflow)
  • Open (5') · "Walk me through what triggers this process and what the final output looks like."
  • Probe (20') · Ask each criterion. Always demand a concrete example or evidence.
  • Close (5') · "Anything about this process about to change in the next 6 months?"
Bias control
  • → Both consultants attend for inter-rater calibration
  • → Do not disclose the 5/6 threshold to interviewees
  • → Each "No" needs a documented gap + mitigation feasibility check
Edge case ·   If a workflow runs <50×/mo but each run takes 8+ hours and carries high error cost, you may keep it as the allowed exception — document the ROI math.

Hard-floor for Design entry: PriorityScore ≥ 3.0 · Target ICC ≥ 0.75 across assessors

Every workflow step is classified as one of five archetypes — Orchestrator · Executor · Analyst · Retriever · Evaluator — making architecture a classification exercise rather than a creative leap

Mutually exclusive · collectively exhaustive · selection trigger codified for each

Coordinator
Core function

  • Routes sub-tasks
  • Maintains state

  • Email / messaging
  • File writers

Phase 2 · Deep-dive2.5 · Three Agent-Human interaction modes

Every step is assigned an interaction mode — Autopilot · Co-Pilot · Guardian — chosen by the worst plausible failure severity, not by team preference

A "trust contract" between the agent and the human; selection is risk-based and visible in the architecture document

Mode 1 · Autopilot

Agent autonomy: Full (execute + log)

When to choose
  • → Low-risk, high-volume, reversible operations
  • → Output verifiable post-hoc by sampling
  • → Recovery cost < review cost
Design implications
  • • Comprehensive logging + audit trail
  • • Post-hoc sampling (5–10%) by reviewers
  • • Anomaly detection threshold for auto-escalation

Example ·  Invoice 3-way match where PO + GR + Inv align within tolerance.

Mode 2 · Co-Pilot

Agent autonomy: Shared (agent proposes, human refines)

When to choose
  • → Medium-risk, judgment-dependent
  • → Output benefits from human refinement
  • → Volume manageable for inline review
Design implications
  • • Draft + diff UI presented to reviewer
  • • Feedback loop captures edits for learning
  • • Time-out → auto-revert or escalate

Example ·  Credit memo drafted by agent; relationship manager edits before send.

Mode 3 · Guardian

Agent autonomy: Limited (agent prepares, human decides)

When to choose
  • → High-risk or irreversible decisions
  • → Regulated / audit-critical outputs
  • → Cost of error >> cost of human review
Design implications
  • • Agent assembles evidence pack, never acts
  • • Decision UI with explainability + sources
  • • Human signature persisted on every decision

Example ·  Sanctions hit triage — agent prepares disposition, compliance officer approves.

Selection rule ·   Start by asking "if the agent gets this wrong, what's the worst plausible outcome?" — Reversible/cheap → Autopilot · Recoverable with rework → Co-Pilot · Irreversible/regulated/financial → Guardian.

Phase 2 · Deep-dive2.6 · Agent-to-agent orchestration patterns

Six A2A patterns cover known multi-agent coordination needs — selection is dependency-based, not preference-based

Pick the simplest pattern that satisfies the dependency structure; complexity is a cost, not a feature

① Sequential

Pipeline
Topology
  • A → B → C
When to use
  • Strict order required
  • Each step needs prior output
Example
  • Retrieve → Analyze → Draft

② Parallel fan-out / fan-in

Concurrent
Topology
  • A → {B,C,D} → E
When to use
  • Independent sub-tasks
  • Latency budget pressured
Example
  • Multi-source data retrieval

③ Hierarchical (Manager-Worker)

Delegation
Topology
  • Manager dispatches → Workers
When to use
  • Dynamic task decomposition
  • Variable sub-task count
Example
  • Research orchestrator → topic agents

④ Negotiation / Debate

Consensus
Topology
  • Multiple agents argue → judge
When to use
  • Quality benefits from challenge
  • Ambiguous best answer
Example
  • Risk assessment + counter-assessor

⑤ Event-driven / Reactive

Pub-sub
Topology
  • Trigger → Listening agents
When to use
  • External signal initiates
  • Decoupled subscribers
Example
  • New SAR triggers triage agent

⑥ Blackboard (Shared state)

Collaboration
Topology
  • Agents read/write shared context
When to use
  • Long-running collaboration
  • Many specialists, shared evidence
Example
  • Multi-discipline diagnostic assistant
Selection rule ·   Linear dependency → Sequential · Independent siblings → Parallel · Dynamic count → Hierarchical · Quality through challenge → Negotiation · Event-initiated → Reactive · Long-running collaboration → Blackboard. Avoid combining patterns until single-pattern fails.

Phase 3 · Deep-dive3.4 · Test plan structure

A solid test plan combines four layers — unit · contract · scenario · adversarial — each producing different evidence the gate needs

Each layer is automated where possible; HITL is reserved for Layer 3 when ODS is low

Layer 1 · Unit

Agent step level
What it tests
  • Tool call correctness
  • Output schema validity
  • Single-step prompt outputs
How automated
  • Pytest / unittest
  • Run on every commit
Pass criterion
  • 100% green

Layer 2 · Contract

Agent ↔ Tool / Agent ↔ Agent
What it tests
  • API payload shape
  • Error path handling
  • Timeout + retry behavior
How automated
  • Mock tool servers
  • Negative-path fixtures
Pass criterion
  • All error codes handled

Layer 3 · Scenario / Eval

End-to-end
What it tests
  • Correctness on golden set
  • HITL escalation triggers
  • Latency & cost
How automated
  • Eval harness · LLM-judge for Q3
  • HITL panel rubric for low ODS
Pass criterion
  • ≥85% on holdout (target)

Layer 4 · Adversarial

Safety & robustness
What it tests
  • Prompt injection
  • Out-of-domain inputs
  • Hallucination / confabulation
How automated
  • Curated adversarial set
  • Red-team prompts
Pass criterion
  • ≥90% safe behavior
Eval harness — minimum capabilities
  • • Replay cases against current agent build · capture trace
  • • Score each case via rubric (rule-based + LLM-judge + HITL where flagged)
  • • Aggregate metrics: correctness, latency, cost, escalation rate, per-class
  • • Diff vs prior build · regression flag
  • • Versioned reports → /reports/<build>.html
Test data discipline
  • • Dev split (70%) — used for iteration · holdout (30%) — locked until Week 3
  • • New edge cases added to dev only · never holdout
  • • "Train on dev, test on holdout" — no exceptions
  • • Holdout pass <5pp below dev = generalization gap acceptable; >10pp = stop, investigate overfitting
